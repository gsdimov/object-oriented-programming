# Предефиниране на оператори

Операторите в C++ са нищо повече от синтактична захар за функции.
Именно това ни позволява да ги предефинираме (тъй като можем да предефинираме функции),
което ще разгледаме по-надолу. Имайте предвид, че операторите могат да връщат произволни
стойности (т.е. спокойно бихме могли да предефинираме `operator+` да връща `void`),
тъй като не можем да различаваме функции само по типа им на връщане, но различните оператори
може да очакват определен брой аргументи (и от определен тип). Освен това има и конвенции,
които е добре да спазваме, за да сме константни и за да получим по-хубави свойства на някои
оператори (напр. верижното извикване на `operator<<`). Забележете, че операторите също
трябва да бъдат константни в определени ситуации.

---

Тук може да намерите подробна информация за операторите в C++: https://www.ibm.com/docs/en/i/7.4.0?topic=only-overloading-operators-c

---

## Оператори, които не могат да се предефинират:

- оператор за достъп на поле/метод (`.`);
- оператор за резолюция (`::`);
- тернарен оператор (`?:`);
- унарен оператор `~`;
- `sizeof`;
- `typeid`;
- `alignof`;
- `decltype`.

---

### Забележка: Всички предефинирани оператори са **методи** на съответния клас освен ако не е указано друго изрично (friend или обикновена външна функция).

---

## Оператори, които могат да се предефинират:

- оператор за индексиране:

  ```cpp
  T &operator[](std::size_t index) {
    // ...
  }
  ```

  ```cpp
  T const &operator[](std::size_t index) const {
    // ...
  }
  ```

  > Горните два оператора са *различни* и компилаторът може да различава между тях,
    тъй като единият е маркиран като неконстантен, а другият - като константен.
    Следователно, когато компилаторът desugar-ва горните дефиниции, така че да използват
    указател към обекта (`this`), то в неконстантния overload указателят this има
    тип `T *this`, а в константния - `T const *this`, които са два различни типа
    параметри на функции.

- унарен оператор `+`/`-`:

  - `operator-` - обикновено се използва за промяна знака на обекта
    (напр. комплексно число) и връща нов обект, без да променя текущия
    (затова е `const`).

  ```cpp
  T operator-() const {
    // ...
  }
  ```

  - `operator+` - обикновено се използва за type promotion
  при числени типове. В курса по СДП ще го разглеждате
  отново, като синтактична захар за позиция в двоично дърво:

  ```cpp
  // това не е важно в този курс
  BinaryTreePosition<T> operator+() const {
    return BinaryTreePosition<T>(currPos->right);
  }
  ```

- побитови оператори `&`, `|`, `~`, `^`, `&=`, `|=`, `~=`, `^=`:

  Най-честият пример за горните е при дефиниране на клас за множество,
  където всеки от горните оператори представя сечение, обединение,
  разлика и симетрична разлика, респективно, като за всеки има
  и съответен *мутиращ* вариант (т.е. не връща нов обект, а променя
  този, който е от лявата страна на оператора).

  Немутиращият вариант връща нов обект на база операцията между
  два обекта (обикновено `*this` и обекта, който се подава като параметър),
  докато мутиращият връща референция към текущия обект, за да може да е
  възможно верижно извикване на оператора, например: `set1 &= set2 |= set3`,
  което се свежда до `(set1 &= set2) |= set3`, тъй като тези оператори
  са лявоасоциативни.

  Честа практика е двойки мутиращ и немутиращ оператор да се имплементират
  един чрез друг (напр. `^` чрез `^=` или `&=` чрез `&`).

  Тук оператор `&` е имплементиран чрез оператор `&=`:

  ```cpp
  Set<T> &operator&=(Set<T> const &other) {
    // логика за сечение
    return *this;
  }
  ```

  ```cpp
  Set<T> operator&(Set<T> const &other) const {
    Set<T> cpy(*this);
    cpy &= other;
    return cpy;
  }
  ```

  Тук оператор `|=` е имплементиран чрез оператор `|`:

  ```cpp
  Set<T> operator|(Set<T> const &other) const {
    // логика за обединение
    return cpy;
  }
  ```

  ```cpp
  Set<T> &operator|=(Set<T> const &other) {
    *this = *this | other;
    return *this;
  }
  ```

  По-често се използва първия вариант (немутиращ чрез мутиращ),
  тъй като се правят по-малко операции на копиране.

  ```cpp
  Set<T> &operator~=(Set<T> const &other) {
    // логика за разлика: *this \ other
    return *this;
  }
  ```

  ```cpp
  Set<T> operator~(Set<T> const &other) const {
    // подобно на operator& и operator&=,
    // но на един ред
    return Set<T>(*this) ~= other;
  }
  ```

  ```cpp
  Set<T> &operator^=(Set<T> const &other) {
    // логика за симетрична разлика
    return *this;
  }
  ```

  ```cpp
  Set<T> operator^(Set<T> const &other) const {
    return Set<T>(*this) ^= other;
  }
  ```

- аритметични оператори:

  Нека имаме следния клас, представящ комплексно число:

  ```cpp
  class Complex {
  // ...
  private:
    double m_real;
    double m_imaginary;
  };
  ```

  Тогава това също е тип, който представя число,
  и бихме искали да можем да извършваме обикновените
  операции с комплексни числа:

  ```cpp
  Complex &operator+=(Complex const &other) {
    m_real += other.m_real;
    m_imaginary += other.m_imaginary;
    return *this;
  }
  ```

  ```cpp
  Complex operator+(Complex const &other) const {
    Complex cpy(*this);
    cpy += other;
    return cpy;
  }
  ```

  Аналогична е имплементацията и на останалите двуместни оператори:

  - `Complex &operator-=(Complex const &other)`;
  - `Complex operator-(Complex const &other) const`;
  - `Complex &operator*=(Complex const &other)`;
  - `Complex operator*(Complex const &other) const`;
  - `Complex &operator/=(Complex const &other)`;
  - `Complex operator/(Complex const &other) const`;
  - `Complex &operator%=(Complex const &other)` (май е безсмислено тук);
  - `Complex operator%(Complex const &other) const`.

  Операторите, които съдържат `=` в себе си (без `==` и `!=`, разбира се),
  винаги променят обекта отляво, така че не могат да са `const`, докато
  тези без `=` не го променят, така че *трябва* да са `const`, подобно на това,
  което знаем от курса по УП:

  ```cpp
  int x = 10, y = 42;
  x += y;  // променя x
  x + y;   // нито една от двете стойности не се променя, връща се нова
  ```

  Технически бихме могли да правим и неща от вида:

  ```cpp
  Complex &operator+=(double x) {
    // логика
  }
  ```

  т.е. не сме ограничени винаги да добавяме обект от тип `Complex const &`,
  макар че в случая това не би имало смисъл (не знаем дали `x` е реална или имагинерна част).

  Би имало смисъл да предефинираме и `operator-`:

  ```cpp
  Complex operator-() const {
    return Complex{-m_real, -m_imaginary};
  }
  ```

  - префиксен/постфиксен оператор `++`/`--`

  Нека б.о.о. си припомним как работят префиксният и постфиксният оператор `++`:

  ```cpp
  int x = 10;

  // първо увеличаваме стойността на x, след което връщаме новата стойност
  int y = ++x; // y = 11, x = 11

  // правим копие на старата стойност, увеличаваме стойността на текущата
  // променлива, след което връщаме копието
  int z = ++x; // z = 11, x = 12
  ```

  Ще искаме да симулираме същото поведение и за обекти.

  Добавянето/изваждането на единица от комплексно число е безсмислено,
  така че ще разгледаме примера за рационално число:

  ```cpp
  class Rational {
  // ...
  private:
    int m_numerator;
    int m_denominator;
  };
  ```

  ```cpp
  // префиксен ++, не приема аргументи;
  // връщаме същия обект, тъй като няма нужда от копие,
  // можем да извикваме ++ верижно: ++(++rat)
  Rational &operator++() {
    m_numerator += m_denominator;  // увеличаваме стойността в текущия обект
    return *this;  // връщаме новопроменения обект
  }
  ```

  ```cpp
  // постфиксен ++, приема фиктивен аргумент (винаги int),
  // за да можем да го разграничаваме с префиксния вариант
  Rational operator++(int) {
    Rational cpy(*this);  // копираме текущия обект
    ++(*this);  // увеличаваме стойността на текущия обект,
                // преизползвайки горната логика за префиксен ++
    return cpy; // връщаме копието от по-горе (стойността преди промяна на обекта)
  }
  ```

  Аналогично имплементираме и `operator--`:

  - `Rational &operator--()` - префиксен;
  - `Rational operator--(int)` - постфиксен.

  И двата случая променяме текущия обект (`*this`), така че не може нито единият
  оператор да бъде `const`. Фиктивният аргумент е нужен, защото не можем да разграничим
  две функции (в частност оператори) само по техния тип на връщане.

- оператори за сравнение:

  Тъй като върху комплексните числа няма еднозначно дефинирана
  линейна наредба, в следните примери ще използваме клас за рационално число.
  Тогава, тъй като всички рационални числа са сравними помежду си,
  би имало смисъл да предефинираме оператори за сравнение:

  ```cpp
  bool operator==(Rational const &other) const {
    return m_numerator == other.m_numerator 
        && m_denominator == other.m_denominator;
  }
  ```

  ```cpp
  bool operator!=(Rational const &other) const {
    return !(*this == other);
  }
  ```

  Аналогично можем да предефинираме и следните оператори,
  като бихме искали да се стремим да преизползваме вече
  реализираните такива, използвайки математически еквивалености,
  например $x < y$ тстк не е вярно, че `x >= y` (това се прави
  с цел максимално преизползване на код и минимизиране на потенциални
  грешки в имплементацията; ако имаме някакви грешки, то тогава би се наложило
  да ги променим само там, където е основната логика, вместо на множество
  различни места):

  - `bool operator<(Rational const &other) const`;
  - `bool operator<=(Rational const &other) const`;
  - `bool operator>(Rational const &other) const`;
  - `bool operator>=(Rational const &other) const`.

- логически оператори:

  Нека си представим, че имаме следния тип, представящ
  израз от съждителна логика:

  ```cpp
  class Proposition {
    // член-данни
  };
  ```

  В действителност изразите от съждителна логика се представят
  най-естествено чрез дървовидни структури, за които все още не сме
  говорили, така че просто приемете, че имаме подходящо представяне.

  Очевидно би имало смисъл да имплементираме конюнкция, дизюнкция, негация и др.:

  ```cpp
  bool operator&&(Proposition const &other) const {
    // логика за конюнкция
  }
  ```

  ```cpp
  bool operator||(Proposition const &other) const {
    // логика за дизюнкция
  }
  ```

  ```cpp
  bool operator!() const {
    // логика за негация
  }
  ```

- оператори за входно-изходни операции с потоци

  В C++ има йерархия на потоците, като е в сила следното:

  - `std::ostream` е най-общият тип на изходен поток;
  - `std::istream` е най-общият тип на входен поток.

  Горното означава, че на неща от тип `std::ostream`
  можем да подаваме както `std::cout`, така и файлови потоци
  и др. Именно това е причината `operator<<` да работи и с двете.

  Затова можем да се опитаме да предефинираме `operator<<` и `operator>>`,
  използвайки горните два типа, под формата на методи на даден клас.
  За целта ще използваме класа за комплексно число като пример:

  ```cpp
  // тук НЕ променяме текущия обект
  std::ostream &operator<<(std::ostream &os) const {
    os << m_real;
    os << (m_imaginary > 0 ? "+" : "");
    os << m_imaginary << "i";
    return os;
  }
  ```

  ```cpp
  // така бихме искали да четем в текущия обект, затова не е const
  std::istream &operator>>(std::istream &is) {
    is >> m_real >> m_imaginary;
    return is;
  }
  ```

  **НО** следните извиквания няма да работят:

  ```cpp
  Complex c{3,64.2};
  std::cout << c;

  Complex cbuf;
  std::cin >> cbuf;
  ```

  Причината за това е, че горните оператори са декларирани като методи,
  т.е. отляво на оператора **винаги** трябва да стои обект на дадения клас
  (т.е. обект от тип `Complex`). Затова би трябвало да ги извикваме така:

  ```cpp
  Complex c{3,64.2};
  c << std::cout;

  Complex cbuf;
  cbuf >> std::cin;
  ```

  което е в разрез с конвенциите за извикване на горните оператори, които вече са ни добре познати.
  Освен че горното е неприятно за четене, по този начин губим и верижното извикване на операторите (защо?).

  Затова тези оператори се правят или обикновени външни функции, или приятелски такива, ако е необходим
  достъп до вътрешното представяне на класа (напр. вектори):

  - ако имаме селектори и мутатори за реална и имагинерна част:

  ```cpp
  std::ostream &operator<<(std::ostream &os, Complex const &complex) {
    os << complex.getReal();
    os << (complex.getImaginary() > 0 ? "+" : "");
    os << complex.getImaginary();
    os << "i";

    return os;
  }
  ```

  ```cpp
  std::istream &operator>>(std::istream &is, Complex &complex) {
    double real, imaginary;
    is >> read >> imaginary;

    complex.setReal(real);
    complex.setImaginary(imaginary);

    return is;
  }
  ```

  - ако нямаме селектори и мутатори за реална и имагинерна част, трябва да имаме
    достъп до private член-данните на класа:

  ```cpp
  friend std::ostream &operator<<(std::ostream &os, Complex const &complex) {
    os << complex.m_real;
    os << (complex.m_imaginary > 0 ? "+" : "");
    os << complex.m_imaginary;
    os << "i";

    return os;
  }
  ```

  ```cpp
  friend std::istream &operator>>(std::istream &is, Complex &complex) {
    is >> complex.m_real >> complex.m_imaginary;
    return is;
  }
  ```

  Друг пример е следният:

  ```cpp
  template<typename T>
  friend std::ostream &operator<<(std::ostream &os, Vector<T> const &vec) {
    for (std::size_t i = 0; i < vec.size(); ++i) {
      os << vec.m_data[i] << ' ';
    }

    return os;
  }
  ```

  Можем да предефинираме оператор за индексиране (или обикновен метод),
  ако не искаме да правим горната функция приятелска.

- други екзотични оператори

  - `operator*` (унарен) - използвали сме го при умни указатели, за да симулираме
    дереренциране на указатели. Използва се също и при итератори и позиции в дървета.

  ```cpp
  T &operator*() {
    return *m_ptr;
  }

  T const &operator*() const {
    return *m_ptr;
  }
  ```

  ---

  - `operator->` - отново, използвали сме го при умни указатели, за да симулираме
  дереренциране и достъпване на поле/метод на обект през подаден указател:

  ```cpp
  T *operator->() {
    return m_ptr;
  }

  T const *operator->() const {
    retur m_ptr;
  }
  ```

  ---

  - `operator T()` - оператор за преобразуване до тип `T`. Например,
  ако имаме следния клас:

  ```cpp
  class IntWrapper {
  private:
    int i;
  };
  ```

  който е просто цяло число, обвито в клас, бихме искали да го третираме като `int`:

  ```cpp
  // това е метод
  operator int() const {
    return i;
  }
  ```

  В действителност вече сте виждали подобен оператор при следното:

  ```cpp
  std::ifstream ifs("file.txt");
  if (!ifs) {
    // ...
  }
  ```

  т.е. по-горе можем да третираме обект от тип `std::ifstream` като булева стойност.
  Това е възможно, защото класът `std::ifstream` има предефиниран `operator bool() const`.

  Не е препоръчително да се предефинират подобни оператори, тъй като това може да доведе до проблеми.
  Например, ако имаме `operator+` за рационални числа, параметризиран конструктор с един параметър (числител)
  и същевременно имаме следната имплементация:

  ```cpp
  operator double() const {
    return static_cast<double>(m_numerator) / m_denominator;
  }
  ```

  То тогава всичко от тип `Rational` може да се сведе до нещо от тип `double` и всичко от тип
  `double` може да се сведе до нещот от тип `Rational`, така че, правейки следното извикване:

  ```cpp
  Rational rat{3,4};
  Rational rat2 = rat + 5;
  ```

  ще получим неяснота (ambiguity) кой оператор трябва да се използва, тъй като има два възможни такива
  заради неявните преобразувания, които могат да се извършат от компилатора:

  1. `double operator+(double, double)`;
  2. `Rational operator+(Rational const &, Rational const &)`.

  ---

  - `T operator()(U args...)` - оператор за извикване на обект като функция. Виждали сме такъв пример
  с клас за полином с реални коефициенти, 
  който сам по себе си представя математичска функция, така че има смисъл да можем да правим следното:

  ```cpp
  Polynomial p;
  std::cout << p(7); // x = 7
  ```

  ```cpp
  // това е метод
  double operator()(double x) const {
    // логика за оценка на полинома в точка
  }
  ```

  За полином с две променливи бихме написали следния оператор:

  ```cpp
  double operator()(double x, double y) const {
    // логика
  }

  Polynomial2 p;
  p(6, 7);
  ```

  ---

  Има и различни други оператори, като например `operator->*` (при достъпване на член-функции през указател),
  `operator<=>` (C++20), `operator new`, `operator new[]`, `operator delete`, `operator delete[]` и т.н.

## Кога има смисъл нещо да бъде приятелска функция

Единият случай е по-горе описаният проблем с `operator<<` и `operator>>`.

Има и още един основен случай, в които бихме искали да използвам външни или приятелски функции.
Нека си представим, че класът за комплексно число има следния конструктор със стойности на параметрите
по подразбиране:

```cpp
class Complex {
public:
  Complex(double real=0, double imaginary=0)
  : m_real(real), m_imaginary(imaginary) {}
};
```

Тогава можем да направим следното без проблем:

```cpp
void foo(Complex const &);
foo(4);
```

тъй като компилаторът ще преобразува стойността 4 до `Complex(4)` *неявно*.

Тогава е логично да се запитаме дали можем да използваме и по-горните оператори по подобен
начин (т.е. има подаваме стойности, които могат неявно да се преобразуват до комплексни числа):

```cpp
Complex c(3.14);

Complex c2 = c + 5;
Complex c3 = 5 + c;
```

И `c2`, и `c3` трябва да работят и да имат еднакви стойности, защото операцията е комутативна.
Но в действителност, имайки само горната дефиниция `Complex operator+(Complex const &other) const`,
би работил само редът за `c2`, тъй като отляво на `+` стои обект от тип `Complex`.
Но в случая на `c3` отляво стои интегрална стойност, която не е непременно обект от тип `Complex`
и в тази ситуация не може да бъде неявно преобразувана до такава.

Един начин да се справим с горния проблем, е да предефинираме оператора и в двете посоки:

```cpp
// като МЕТОД
Complex operator+(Complex const &other) const {
  // логика
}

// като НЕПРИЯТЕЛСКА ВЪНШНА ФУНКЦИЯ
Complex operator+(double lhs, Complex const &rhs) {
  return Complex(lhs) + rhs;
}
```

Разбира се, бихме могли да използваме и приятелска функция,
но в този случай би било излишно нарушаване на принципа на капсулация.

Така вече горният ред за `c3` би работил, като би извиквал функцията
`Complex operator+(double, Complex const &)`.

Друг начин е да забележим, че неявни преобразувания могат да се извършват при подаване
на аргументи (както при `c2`). Тоест, е достатъчно да накараме `operator+` да приема
и двата си операнда като аргументи:

```cpp
Complex operator+(Complex const &lhs, Complex const &rhs) {
  return Complex(lhs.getReal() + rhs.getReal(),
                 lhs.getImaginary() + rhs.getImaginary());
}
```

Разбира се, горната функция е външна и също потенциално би могла да бъде приятелска.

По горния начин обединяваме и двете посоки за `c2` и `c3` наведнъж. Ако бихме искали да
имаме такива неявни преобразувания, бихме искали да използваме този подход, например:

- `c == 5.1`;
- `6 != c`;
- оператори за линейна наредба, ако има смисъл (идеята е същата);
- други аритметични оператори.

Например,

```cpp
bool operator==(Complex const &lhs, Complex const &rhs) {
  return lhs.getReal() == rhs.getReal()
      && lhs.getImaginary() == lhs.getImaginary();
}
```

което би ни позволило неща от рода на `c == 2` и `2 == c`.

Забележете, че поне единият аргумент трябва да бъде от тип `Complex`.
